## 1. 项目背景与业务价值（讲给业务听众）

### 1.1 解决什么问题
物联网项目（无人机巡检、智能驾驶、水务抄表、车载监控、共享设备……）会用到海量蜂窝物联网卡，且往往**跨三家运营商**。痛点是：

- 卡分散在移动 OneLink、电信 CMP、联通雁飞智连三个平台，**各查各的**，没有统一视图；
- 卡的**状态轮询、续费、停复机、流量/语音池**要在不同平台手工操作；
- 设备（守护码 ↔ ICCID 映射、激活、心跳、告警）与卡**没有打通**；
- **财务**（订单、分账、对账、风控）和**数据**（在线率、续费率、ARPU）缺乏统一沉淀。

### 1.2 平台给出的价值（对应需求文档 5 大版块）

| 版块 | 业务价值 | 关键能力 |
|------|---------|---------|
| ① 连接引擎 | 通信资源统一管理与风控 | 多运营商接入、ICCID/IMEI/eSIM 统一管理、状态轮询、流量池/语音池、套餐生命周期、续费/变更/停复机、白名单/风控 |
| ② 设备管理 | 设备全生命周期 | 型号库、批量入库/发货/激活、守护码↔ICCID 映射、心跳、告警、远程参数下发 |
| ③ 服务版块 | 场景服务订阅与体验 | 套餐包/服务包、服务有效期、续费/升级/停服、异常通知、客服协同、SLA |
| ④ 财务管理 | 收入、分账与风控 | 支付/订单/退款、渠道分佣、分账结算、发票/对账、交易风控、价格策略 |
| ⑤ 数据版块 | 分析、决策与监控 | 在线率、在网率、续费率、ARPU、风控监控、设备分布热力 |

> 演示话术：「这套平台把**运营商能力、设备、服务、钱、数据**五件事收敛到一个后台。今天我先带大家走一遍界面，再让大家看到它**真的能调通三大运营商的接口**。」

## 2. 系统架构（讲给技术听众）

### 2.1 总体结构

```
┌──────────┐    HTTP     ┌────────────────────┐    HTTPS+鉴权   ┌─────────────────────┐
│  浏览器   │  ───────►   │  前端 (Umijs Max)   │                 │ 移动 OneLink (雁飞)  │
│ 管理人员  │   :8000     │  物联网     │                 │ 电信 CMP / 5GCMP     │
└──────────┘             │  React19 + antd5    │                 │ 联通 雁飞智连        │
                         └─────────┬──────────┘                 └──────────▲──────────┘
                                   │ REST /api                              │
                                   ▼                                        │
                         ┌────────────────────┐   运营商网关签名/Token      │
                         │  后端 (NestJS 11)   │  ──────────────────────────┘
                         │  :3000  /api 前缀   │
                         │  Swagger /docs      │
                         └─────────┬──────────┘
                                   │ Mongoose
                                   ▼
                            ┌────────────┐
                            │  MongoDB   │
                            └────────────┘
```

### 2.2 技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 前端 | 物联网 6 + UmiJS Max 4 + React 19 + antd 5 | 企业级中后台脚手架，自带布局/权限/国际化/请求库 |
| 后端 | NestJS 11 + TypeScript 5 | 模块化、依赖注入、`@nestjs/swagger` 自动生成接口文档、`class-validator` 参数校验 |
| 持久层 | MongoDB（优先）/ 进程内存（兜底） | 自研 `StoreService` 启动时探测 Mongo，连不上自动回退内存（演示零数据库依赖）；统一 `BaseCrudService` 派生各业务模块 |
| 配置 | `@nestjs/config` | `.env` 全局注入，移动接口支持沙箱/生产切换 |
| 运营商对接 | 自研 Gateway Service | 各运营商一个 `*-gateway.service.ts` 负责签名/Token，业务 `*.service.ts` 负责组装与解析 |

## 4. 演示脚本 A：全栈真实闭环走查（业务主线，约 8–10 分钟）

## 5. 演示脚本 B：运营商真实接口对接（技术亮点，约 5–7 分钟）

> 平台不止有自有业务数据，还**真实对接了运营商**。两种演示方式：① 「卡标识管理」页点「**同步移动卡(真实接口)**」直接触发；② Swagger 现场调。
> ⚠️ 现场若运营商沙箱不可达，会返回网关错误（HTTP 502）——这恰恰**证明后端真的在向运营商发起请求**；沙箱可达时即返回真实卡信息。

### 步骤 1 · 打开 Swagger
- **点**：http://localhost:3000/docs
- **说**：「后端用 NestJS，接口文档由代码自动生成。除了卡/设备/订单等业务模块，看 `SIM 卡` 这一组——它编排了不同运营商的真实查卡。」

### 步骤 2 · 移动：批量查询码号信息
- **接口**：`POST /api/sim/mobile/card-info/batch`
- **说**：「这对接的是**移动 OneLink（雁飞）**的『码号信息批量查询』。我们的服务会先调 OneLink 拿 Token，再带签名查卡，msisdns / iccids / imsis 三选一。」
- **演示**：点「Try it out」，填入示例：
  ```json
  { "iccids": "898604xxxxxxxxxxxxx,898604yyyyyyyyyyyyy" }
  ```
  Execute → 展示返回结构（`traceId / msg / data[]`）。
- **技术点**：
  - 沙箱/生产由 `.env` 的 `MOBILE_ONELINK_USE_SANDBOX` 一键切换，**不改代码**；
  - 多值入参自动把逗号转下划线（贴合 OneLink 报文要求）；
  - Token 获取、transid 生成都封装在 `china-mobile-gateway.service.ts`。

### 步骤 3 · 电信：批量查询 SIM 卡资料
- **接口**：`POST /api/sim/telecom/card-info/batch`
- **说**：「这对接**电信 5G 连接管理平台 CMP** 的 `CTIOT_5GCMP_BQ018`。请求走网关，自动加 **AppKey / Sign / Timestamp** 签名，单次最多 100 个接入号。」
- **演示**：填 `accessNumbers`（接入号数组）+ `custNumber`（客户编码，`.env` 已配默认值）→ Execute。
- **技术点**：参数校验（接入号数量上限、必填项）由 `class-validator` 在进入业务前拦截，错误会返回 400。

### 步骤 4 · 联通 & 架构收口
- **说**：「联通『雁飞智连』的网关鉴权（app_id/timestamp/trans_id/token，SHA-256 签名）也已经落在 `china-unicom` 模块，业务接口按同一套 Gateway 模式即可快速扩展。三家运营商**同一套架构**：一个 gateway 管签名，一个 service 管业务，对前端暴露统一的 `/api/sim/*`。」

> 收口话术：「所以这个平台**前端是完整的产品界面，后端是真实的运营商对接能力**，两边按统一 REST 协议拼起来，就是一个可上线的商用版本。」
